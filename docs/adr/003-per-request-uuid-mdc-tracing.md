# ADR-003: Per-Request UUID Tracing via Servlet Filter and SLF4J MDC

**Status:** Accepted
**Date:** 2026-03-14
**Feature:** MTI Scores API - Product Requirements Document

## Context

The PRD requires a `request_id` UUID v4 field in every response (both success and error) and specifies that the request ID should be logged with all operations for traceability (section 8, note 5). Without a consistent tracing mechanism, correlating log lines across the filter chain, service, and repository for a single request is impractical. Options considered were: (1) generate `request_id` in the controller only (too late — filter-level errors would lack an ID), (2) use Spring's `RequestContextHolder` (tightly coupled to Spring MVC internals), and (3) use a dedicated servlet filter that writes the ID to both the `HttpServletRequest` attributes and SLF4J MDC at the earliest point in the pipeline.

## Decision

Implement **`RequestIdFilter`** (`jakarta.servlet.Filter`, `@Order(1)`) that runs before all other filters. It generates `UUID.randomUUID().toString()` per request, stores it as `HttpServletRequest` attribute `RequestIdFilter.REQUEST_ID_ATTR = "requestId"`, and writes it to `MDC.put("requestId", requestId)`. The MDC entry is cleared in a `finally` block after the filter chain completes to prevent thread-pool leakage. Downstream components (controller, service, exception handler) read the ID from the request attribute; log appenders pick it up automatically from MDC.

## Consequences

- **Easier:** All log statements automatically include `requestId` via MDC without passing it as a method argument to every layer (except the service, which receives it explicitly for inclusion in the `MetaDto` response body). Error responses generated by `GlobalExceptionHandler` can include a valid `request_id` even for exceptions raised outside the service layer.
- **Harder:** The `REQUEST_ID_ATTR` constant is a shared coupling point between `RequestIdFilter`, `VesselController`, and `GlobalExceptionHandler`; changes to the attribute name must be synchronised across all three.
- **Constraint:** Any new servlet filter added in future that needs `requestId` must be `@Order` greater than 1 to ensure `RequestIdFilter` has already run. The `RateLimitFilter` is `@Order(2)` by convention.
